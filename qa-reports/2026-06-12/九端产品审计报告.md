# Dazong 九端产品级审计报告

执行日期：2026-06-12  
事实源：当前工作区代码、FastAPI OpenAPI、隔离 QA 数据库、Vue Router、三个 Android Debug APK  
结论：**当前不建议按生产级标准验收或直接对公网开放。**

## 一、总体结论

本轮共登记 33 项问题：

| 等级 | 数量 | 结论 |
|---|---:|---|
| P0 | 4 | 存在任意文件读取、跨角色全量订单、匿名图片访问、通用默认密码 |
| P1 | 12 | 存在非法状态推进、资金口径冲突、幂等失败、核心页面被第三方阻塞等 |
| P2 | 14 | 字段校验、异常恢复、移动端安全和跨端一致性不足 |
| P3 | 3 | 自动化和实机验收覆盖不足 |

系统主流程已经具备较完整的业务骨架，六类 PC 页面、三个 Android 工程和主要 API 均能运行，但当前风险集中在四个方面：

1. **权限边界不完整**：工厂账号可通过通用订单接口看到 198 张无关订单；供货商可看到 17 份全局合约。
2. **状态机与账务不一致**：已取消车次可以跳过前序站点确认送达；客户结算后旧 Bill 全部显示已结算，而供货商账期单仍待确认。
3. **安全默认值不适合生产**：演示账号、司机密码、匿名 MinIO、Vite 文件服务和移动端固定地址均可形成真实攻击面。
4. **异常处理会制造“假成功/假空数据”**：移动端断网仍显示“系统在线/暂无数据”，就绪探针在 MySQL 停机时仍返回 200。

## 三、最高风险问题

### P0

1. **Vite 演示控制台任意文件读取**  
   请求 `/demo-console/%2e%2e/%2e%2e/etc/passwd` 返回真实 `/etc/passwd`。根因是解码后直接 `join(DEMO_DIST, urlPath)`，未校验最终路径仍位于根目录内。  
   证据：`evidence/security/vite-traversal.json`

2. **工厂账号可查看全量订单**  
   工厂专用接口返回 0 张订单，但同一工厂令牌调用通用 `/api/orders` 返回 198 张订单。通用列表只过滤 client、delivery、supplier，没有 factory 分支。  
   证据：`evidence/security/dynamic-security-checks.json`

3. **投诉、质检和收货照片可匿名访问**  
   MinIO 直链和后端代理均无需登录返回 200，测试对象最大约 4.2 MB。  
   证据：`evidence/security/minio-anonymous-access.json`

4. **生产身份使用通用演示密码**  
   六类 PC 演示账号均可登录；所有激活车辆统一接受 `demo123`，司机端还直接展示该密码。运营创建账号未传密码时也默认使用 `demo123`。  
   证据：`evidence/api/login_results.json`、`evidence/security/dynamic-security-checks.json`

### P1

1. **已取消车次可跳站送达**  
   车次 42 状态“已取消”，第 1 站仍“待发车”，直接提交第 5 站送达返回 200，并把订单推进到“收货”。接口没有校验车次状态和前序站点。  
   证据：`evidence/android/driver-state-checks.json`

2. **空订单、零数量和负数量订单均可创建**  
   负数量订单生成 `total_amount=-1.50`、`total_weight_kg=-2.0`。  
   证据：`evidence/api/api_targeted_tests.json`

3. **重复下单无幂等保护**  
   两个并发相同请求生成两张不同订单。  
   证据：`evidence/api/api_targeted_tests.json`

4. **账务结算口径冲突**  
   客户结算后 8 条旧 Bill 全部“已结算”，但配送对供货商的 3 条应付和供货商 3 条应收 Statement 仍“待确认”。  
   证据：`evidence/database/settlement-consistency.json`

5. **收货幂等并发返回 500**  
   相同 Idempotency-Key 同时请求，一次 200、一次 500；数据库实际已入账。失败原因是账期唯一键冲突，客户端重试会无法判断真实结果。  
   证据：`evidence/database/receive-concurrency.json`

6. **配送车辆页被北斗接口阻塞**  
   `/api/delivery/vehicles` 在 5 秒内无响应并超时，而健康接口仍在 12 ms 内响应。核心车辆管理页不应同步等待第三方。  
   证据：`evidence/security/runtime-resilience.json`

7. **就绪探针不能反映依赖状态**  
   MinIO、MySQL 分别停机时 `/readyz` 均返回 200；MySQL 停机时真实鉴权请求返回 500。  
   证据：`evidence/security/dependency-readiness.json`

其余 P1 包括供货商全局合约、公开账号枚举、JWT 查询参数日志、智能秤假在线/演示读数和 Android 发布签名问题，详见缺陷清单。

## 五、VPN 与第三方结论

VPN 本身不是业务缺陷，但当前代码把第三方可用性直接传导到核心页面：

- 北斗实时数据调用可拖死车辆列表。
- Android 固定 ngrok 地址，地址失效时三个端均受影响。
- 移动端断网或接口超时时保留“暂无数据/系统在线”，用户无法区分真实空数据和网络失败。
- `/readyz` 不检查 MySQL、MinIO，因此容器编排无法正确摘除故障实例。

建议所有高德、北斗、萤石、OCR、AI、打印和温控调用统一采用短超时、缓存、熔断、明确错误状态和非阻塞降级。

## 七、未覆盖与实机验收

以下项目不能由模拟器判定通过：

- 智能秤串口协议、稳定值、去皮、断连和权限。
- UVC 相机抓拍、照片方向、存储失败和双签现场身份。
- PDA 扫描头广播、连扫节流、不同条码制式和标签打印。
- 司机 GPS 漂移、后台定位、权限永久拒绝、高德 Key 白名单。
- 精创、萤石、快麦、北斗真实设备和云台动作。

这些项目必须使用真实硬件、真实网络和最小受控第三方账号完成最终验收。

## 八、修复顺序

1. 立即关闭公网演示入口、匿名 MinIO、默认密码和 Vite 文件穿越。
2. 修复工厂订单、供货商合约、司机送达的服务端授权与状态机。
3. 修复订单数量/金额校验、下单幂等、收货并发和账务结算一致性。
4. 将第三方调用移出核心同步路径，补真实 readiness。
5. 修复移动端错误态、密码框、发布签名、固定地址和网络安全配置。
6. 完成实机验收后再进行生产验收。

完整复现步骤见 `缺陷清单.csv`，覆盖情况见 `测试矩阵.md`。
