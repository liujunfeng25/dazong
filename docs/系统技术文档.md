## 1. 系统定位

系统服务于食堂大宗物资采购与履约，围绕采购方、供货方、配送商、指定厂家、运营方和监管方六类角色，连接智能秤、分检 PDA 和司机端三个 Android 终端。

系统覆盖：

- 招投标、合约和分类浮动价格。
- 食堂维度下单、OCR/语音辅助录入和订单追踪。
- 配送商多供货方分单、供货出库和标签。
- PDA 分检、路线规划、调度车次、装车和司机送达。
- 智能秤逐行称重、拍照、退货留痕和双签收货。
- 批次或周期质检、投诉工单、账期和多方对账。
- 监管态势、审计链、广播、价格数据、预测和 AI 问答。

## 2. 角色与终端

### 2.1 六类业务角色

| 角色 | 身份边界 | 主要入口 |
| --- | --- | --- |
| 采购方 `client` | 登录后还必须选择一个启用食堂，JWT 带 `canteen_id` | `/client/dashboard` |
| 供货方 `supplier` | 只能查看分配给自己的订单行和对应金额 | `/supplier/orders` |
| 配送商 `delivery` | 合约履约、分单、仓储、车辆、调度和配送责任主体 | `/delivery/workbench` |
| 指定厂家 `factory` | 只处理标记为指定厂家且分配给本厂家的商品行 | `/factory/orders` |
| 运营方 `operation` | 基础资料、合约订单、工单账期和识别训练管理 | `/operation/dashboard` |
| 监管方 `monitor` | 跨主体只读监管、预警、广播、分析和 AI 助手 | `/monitor/dashboard` |

`factory` 不是独立的加工生产系统。当前代码没有进厂登记、加工任务、成品入库等工厂工单模型。

### 2.2 三个 Android 终端

| 终端 | 登录身份 | 主要能力 |
| --- | --- | --- |
| 智能秤 | 采购方账号并选择食堂 | 串口读秤、UVC 拍照、逐行确认、退货照片、双签收货 |
| 分检 PDA | 配送商账号 | 当日分单列表、标签扫码、重复/异常判断、分检进度 |
| 司机端 | 车辆车牌和司机密码，签发独立 driver token | 当日车次、站点商品、路线地图、外部导航、确认送达 |

### 2.3 辅助模块

- `demo-console/`：开发和演示造数、清理、推进链路。
- 天枢大屏：监管沉浸式三维展示。
- BI 与价格驾驶舱：监管数据分析视图。
- 运营图像识别：智能秤图片样本、ROI、训练、发布和模型部署。

这些模块不增加数据库角色数量。

## 4. 核心业务流程

### 4.1 招投标与合约

1. 采购方创建招标，确定履约周期、分类范围和价格浮动规则。
2. 配送商查看招标并提交报价。
3. 采购方评标并授标。
4. 系统生成状态为“已中标”的合约。
5. 下单时按食堂、配送日期和分类范围校验有效合约。

合约状态只有“招标中、已中标、已过期”。

### 4.2 下单与食堂隔离

1. 采购方登录。
2. 调用 `GET /api/client/canteens` 获取可用食堂。
3. 调用 `POST /api/client/canteen-session` 换取带食堂上下文的新令牌。
4. 在 `/client/orders/new` 搜索合约范围内商品并下单。
5. 订单保存食堂、配送商、配送地址、经纬度、日期、时窗、重量和体积等信息。

OCR 和语音只负责辅助形成订单行，最终仍由用户检查后提交正式订单。

### 4.3 分单、供货与分检

1. 新订单进入配送商待分单阶段。
2. 配送商通过智能分单预览规则和候选供货方，再提交分单结果。
3. 指定厂家商品只能分给其配置的指定厂家。
4. 普通供货方或指定厂家查看分给自己的订单行、打印分单标签并出库。
5. 分检 PDA 扫描标签，确认分单属于当前配送商、当日订单且已出库。
6. 每个分单只允许生成一条有效分检记录，重复扫描返回重复提示。

### 4.4 路线、车次与司机

1. 配送商按计划日期、车辆能力、订单时窗和地理位置生成路线建议。
2. 路线提交后形成调度车次、站点和随车商品记录。
3. 调度员逐项确认装车；存在未就绪商品时正常发车会被阻止。
4. 必要时可记录原因和照片后执行异常发车。
5. 调度端发车后，司机端才能开始配送。
6. 司机确认站点送达，订单从“发货”进入“收货”。
7. 全部站点送达后车次完成。

路线建议是计算结果；提交后的 `DeliveryDispatchTrip`、`DeliveryDispatchStop` 和 `DeliveryDispatchItem` 才是持久化调度事实。

### 4.5 智能秤收货与退货

1. 智能秤以采购方账号登录并选择食堂。
2. 终端读取订单和收货行。
3. 串口读取重量，锁定时通过 UVC 摄像头保存商品照片。
4. 用户按订单行确认实收数量或重量，并填写少收/退货原因。
5. 退货照片和双签图片上传到订单收货记录。
6. 完成收货后订单从“收货”进入“收货确认”。
7. 若配送商启用了退货复核，相关退货需在结算前审核。

### 4.6 质检、投诉与账务

- 商品可配置批次质检或周期质检。
- 供货方和指定厂家提交对应报告，运营方审核周期报告及修订。
- 采购方可针对订单发起投诉，配送商回复，运营方结案。
- 工单状态为“待处理、处理中、已关闭”。
- 账期生成各主体的应收/应付账单和食堂维度对账单。
- 对账确认、结算和订单最终结算分别由明确接口控制。

### 4.7 监管与 AI

- 监管端聚合订单、履约、车辆、设备、温湿度、异常、质检和账务。
- 审计日志、状态日志、Outbox 事件和通知用于追踪关键动作。
- 监管广播可按全域、角色或用户发送，并记录接收和已读状态。
- AI 助手使用业务查询工具回答实时数据问题，使用操作手册 RAG 回答“怎么操作”问题。
- 无模型密钥或模型网络失败时，可降级为本地工具和规则回答；页面必须显示降级事实。

## 5. 状态模型

### 5.1 主订单状态机

```text
下单 -> 配货 -> 发货 -> 收货 -> 收货确认 -> 已结算
  \-> 取消
```

合法流转：

| 当前状态 | 允许的下一状态 |
| --- | --- |
| 下单 | 配货、取消 |
| 配货 | 发货 |
| 发货 | 收货 |
| 收货 | 收货确认 |
| 收货确认 | 已结算 |
| 取消 | 无 |
| 已结算 | 无 |

### 5.2 配送派生阶段

配送阶段由订单状态、分单数量、已出库数量和 PDA 分检记录共同计算：

| 阶段 | 判定与动作 |
| --- | --- |
| 待分单 | 下单/配货状态且没有分单，进入智能分单 |
| 待供货商发货 | 已分单但未全部出库，可先安排路线 |
| 待分拣 | 全部分单已出库但未全部扫码 |
| 待取货 | 分检完成，等待装车和发车 |
| 配送中 | 主订单状态为“发货” |
| 待客户收货 | 主订单状态为“收货” |
| 待结算 | 主订单状态为“收货确认” |
| 已完成 | 主订单状态为“已结算” |
| 已取消 | 主订单状态为“取消” |

派生阶段不单独存入订单状态字段。

## 6. 数据领域

主要持久化领域包括：

- 身份与组织：`User`、`ClientCanteen`。
- 商品与采购：`Category`、`Product`、`Tender`、`TenderItem`、`TenderBid`、`Contract`。
- 订单履约：`Order`、`OrderItemAllocation`、订单状态日志、收货行、退货和评价。
- 配送物流：车辆、设备绑定、地理围栏、仓库、冷链设备、调度车次、站点、随车商品、分检记录。
- 质量与售后：质检报告、周期质检报告、工单和告警。
- 财务：账期、账单、账单汇总和对账单。
- 监管与事件：审计日志、通知、监管广播、Outbox 事件和 IoT 数据。
- 图像识别：类别、样本、ROI、训练任务、模型包和发布记录。

历史订单使用商品和分类快照，避免基础资料修改后破坏追溯口径。

## 8. 外部集成

| 集成 | 用途 | 失败时行为 |
| --- | --- | --- |
| 高德地图 | 地理编码、路径和地图展示 | 显示错误或降级路线，不影响订单基本操作 |
| 北斗 18GPS | 车辆实时位置和历史轨迹 | 保留最后状态并显示接口错误 |
| 萤石云 | 车辆/仓库摄像头直播和云台 | 不影响订单履约 |
| 精创冷云 | 仓库温湿度实时与历史数据 | 显示设备离线或服务不可用 |
| 百度表格 OCR | 采购单图片解析 | 未配置时使用明确标识的演示解析 |
| DashScope | AI 规划、回答和 Embedding | 降级到本地规则/工具，不能伪装为模型成功 |
| 新发地 | 北京市场行情抓取和预测训练 | 使用已有数据，记录超时和限流 |
| 农产品价格网 | 全国价格采集、质量校验和分析 | 保留抓取日志和数据质量标记 |
| 快麦打印 | 分单标签云打印 | 打印失败不得自动标记出库 |
| MinIO | 图片、附件、报告和 APK 下载 | 上传失败阻止依赖附件的提交 |

VPN、代理、DNS、白名单和供应商限流都可能导致外部接口超时。排障时应区分“业务代码错误”和“当前网络无法访问供应商”。

## 9. 部署与运行

### 9.1 当前基线

- 开发：Docker Compose `dev` profile，前端源码挂载，后端源码挂载。
- 生产：Docker Compose `prod` profile，前端 Nginx 静态资源，后端镜像运行。
- 数据库：MySQL 8。
- 对象存储：MinIO。
- 后端镜像：Python 3.11。
- 前端构建：Node.js 20。

当前仓库没有 K8s、Helm、Ingress、Service 或 Deployment 清单。

### 9.2 启动

```bash
cp .env.example .env
make dev-up
```

生产：

```bash
make prod-up
```

### 9.3 生产要求

- 使用随机强密码和独立 JWT 密钥。
- `DEMO_MODE=false`、`SEED_ON_START=false`。
- 根据迁移策略决定是否允许 `AUTO_CREATE_SCHEMA`。
- 限制 MySQL、MinIO 和后端端口的公网暴露。
- 配置 HTTPS、域名、CORS、对象存储公开地址和备份。
- 第三方密钥只写入部署环境，不写入仓库或正式文档。

## 10. 运维与可观测性

- 存活检查：`GET /api/system/healthz`。
- 就绪检查：`GET /api/system/readyz`。
- HTTP 响应包含 `X-Trace-Id`，便于跨日志关联。
- 订单状态、重要操作、通知和异步事件分别有日志或持久化记录。
- 后台任务包括超期扫描、账务跟进和可选 IoT 模拟器。
- 迁移文件当前到 `20260611_0039_smart_scale_roi_review.py`。

详细操作见 [运维手册](ops-runbook.md)。

## 11. 已知限制

- 司机登录密码目前仍是演示逻辑，生产化前必须改成独立凭据管理。
- 外部地图、行情、AI、摄像头和 IoT 服务受网络、VPN、供应商配额和白名单影响。
- `AUTO_CREATE_SCHEMA` 与 Alembic 同时存在，生产应确定唯一受控的结构变更流程。
- Android 三个目录当前未纳入独立 Git 仓库，版本追踪和发布流程需要额外治理。
- 项目当前只有 Docker Compose 部署资料；若未来采用 Kubernetes，必须以实际清单另建部署章节。
